# 注册表(registry)

`urtext index` 会扫描你的规范，并把它们协调进工作区根目录下的
`.urtext/registry.sqlite`。注册表不是缓存 —
它是一条**不可变修订(revision)链**，这种不可变性让 Urtext 能在不相信任何人记忆的情况下
回答关于历史的问题。

## 每个文件一条修订链

每个规范文件都有自己的链，以 `(spec_path, revision)` 为键：

- 每个修订记录文件的 `content_hash = sha256:<hex>`。
- **未变化的内容是无操作。** 重新索引未触碰的文件不会添加任何东西。
- **已变化的内容会追加一个新修订** — 要么是 `ready`（干净），要么
  是 `building`（有错误）。

不变量是绝对的：**历史修订永远不会被覆盖。**
这是存储层对[向前流动
持久化](../guides/04-persistence-model.md)的表达 — 已完成修订是
永久记录，而不是可编辑的草稿区。

> **v0 范围：删除。** 注册表*有*一个 `tombstoneFile` API，会追加一个
> 内容哈希为空的 `tombstoned` 修订，但扫描器只枚举
> 仍然存在的文件，并且从不调用它（`src/scanner.ts`）。所以在 v0 中，删除
> 一个规范文件会让它的最后一个修订继续存活，而不是为它立墓碑 —
> tombstone 路径已经构建，但尚未接入 `index`。

## 两个哈希，两个不同问题

除了文件的 `content_hash`，子句(clause)还携带第二个哈希：

- `content_hash` 回答“*文件*是否改变？” — 它覆盖每一个字节。
- `text_hash = sha256(heading + body)` 是一个**正文哈希** — 它覆盖
  子句的标题和正文文本，并且*排除*锚点元数据。

这个区别决定什么会触发下游失效。编辑锚点
字段会改变 `content_hash`，但**不会**改变 `text_hash`，所以它不会传播
陈旧状态。仔细读这一点：它意味着对 `oracle`、`oracle_ref`、
`risk` 或 `refs` 的修改 — 这些修改可能实质性改变一个子句如何被验证，或它
依赖什么 — **不会**让下游证据(evidence)失效，因为它们存在于
锚点中。`text_hash` 跟踪正文，而不是完整语义。只有对子句*文本*的修改
才会通过[链接器(linker)](04-linker-impact.md)传播陈旧状态。

## 引用随链一起版本化

子句声明的每条 `refs` 边都会存入 `clause_refs` 表，并与
修订链一起版本化。每次扫描后，链接器都会根据
**整个工作区最新的活动修订**解析引用 — 不只是
正在被索引的文件。这就是悬空引用能被捕获的原因：如果文件 A
指向 `B#C003`，而有人在不触碰 A 的情况下从 B 删除了 `C003`，只有一次
全工作区遍历才能看到这个断裂。因此该检查属于 `check`
时刻，而 `unknown_ref` 不会静默腐蚀 A 自己的修订状态 — 它会
作为跨工作区的验证失败报告出来。

## 证据是追加式，失效是一个标志

注册表对可变性的唯一让步狭窄且有原则。当一个
上游子句的 `text_hash` 改变时，它的反向依赖闭包中每个子句的
现有证据都会被盖上 `invalidated_at` — 这是证据表中唯一的
可变列。证据**永远不会被删除**；它被标记为作废，
并保留下来用于审计。Urtext 中没有任何东西会抹掉
曾经检查过什么以及如何检查的记录。

链就位后，运行 oracle(判据) 并记录其判定结果就是
[验证器](03-verifier.md)的工作。
