# 编写子句(clause)

写好子句是一门手艺，Urtext 坦诚地承认，这门手艺并不能
完全变成一本规则书——规范该切多细，部分取决于品味。真正
*可以*传递的是一组你可以有意做出的决策，而不是凭反射行事。
本指南讲的就是这些决策。

> 关于诚实的一点说明：这门手艺能否被教授，而不只是凭感觉掌握，是
> [项目正在跟踪的开放问题](05-adoption-and-limits.md)，不是已经解决的问题。
> 请把下面内容当作决策树，而不是保证。

## 说明性文字还是子句？

这是第一个、也是最重要的决定。**不是所有内容都应该成为子句。**

- 为上下文、动机、背景，以及未来读者理解*为什么*所需的一切编写
  **说明性文字**——这些都不会被检查，而这正是正确的。
- 只有规范性陈述才写成**子句**：系统*必须*满足的东西，
  其违反就是你希望看到红色标记的缺陷。

测试标准：*如果这被违反，我会希望构建失败吗？* 如果是，它就是
子句，并且必须绑定 oracle(判据)。如果不是，它就是说明性文字。粒度随
风险而变化——资金路径值得细粒度子句；UI 细节可能一个也不需要。
过度规范化本身就是一种失败模式：它会抬高子句与代码比例，并
把 Urtext 变成官僚流程。

## 选择 oracle 类型

一旦某件事成为子句，就选择你实际能写出的最强 oracle：

| 如果需求是…… | 使用 | 示例 ref |
|---|---|---|
| 带输入/输出的行为规则 | `test` | `tests/coupon-stack.test.ts` |
| 可以通过运行命令检查 | `cmd` | `scripts/check-migrations.sh` |
| 数值阈值 | `metric` (v1) | `p99<200ms` — *v0 中不可执行* |
| 对可变更范围的边界 | `diff-scope` | `src/billing/**` |
| 确实只能由人判断 | `manual` | 要检查内容的描述 |

优先选择 `test` 和 `cmd`——它们会产生**客观、可重跑**的证据(evidence)。
（独立性不是 Urtext 强制执行的事情——它记录的是 oracle 的 kind 和
ref，而不是谁写了它；让检查真正独立于实现，
是你的纪律，背后由元审计(meta-audit)兜底。）最后才使用 `manual`，并且要知道它的
占比是被跟踪的健康指标：一个大多是 `manual` 的规范，
也大多不是可判定的。

## 避免作弊 oracle

元审计存在就是为了捕捉这种失败模式，但你应该避免它们出现在
源头：

- **`cmd:true`**——总是以零退出的命令。绿色，但什么也不检查。
- **没有任何断言的 `test`**——它会运行、会通过，只能证明代码
  没崩溃，除此之外什么也证明不了。
- **宽到永远不会被违反的 `diff-scope`**——`**` 允许一切。

这些形式上都是绿色，语义上却是空的。
[元审计](../mechanisms/06-meta-audit-gate.md)是安全网——一个
不同模型的审计者会问“这份证据真的覆盖了该子句吗？”——但
这张网是后备，不是许可证。写下那种 oracle：
如果由别人写出，你也会信任它。

## 有意设置风险

`risk:low` 是默认值，它表示“绿色证据*加上*一次元审计 `agree`
可以自动通过”（见[裁决门(gate)](../mechanisms/06-meta-audit-gate.md)）。把
`risk:high` 留给那些代码仍是唯一可评审事实的路径——安全、
资金、迁移、并发、不可逆操作。`risk:high` 子句不会
仅凭绿色证据自动通过；它会路由到人工评审
（[不安全通道(unsafe lane)](../mechanisms/07-unsafe-lane.md)）。把所有东西都标成 `high`
会破坏[裁决门](../mechanisms/06-meta-audit-gate.md)本来要达成的收敛；
把资金路径标成 `low` 则会隐藏这个抽象本应保持可见的
危险。

## 用 refs(引用) 让影响真实可见

当一个子句依赖另一个子句时，用 workspace 相对路径声明它：
`refs:specs/billing/spec.md#C003`（从 repo 根目录写，而不是
`billing/spec.md#C003`）。这
不是文档——正是它让 `urtext impact` 能机械地回答“这个变更
会触及什么？”，并在上游子句含义变化时驱动
失效传播。一个你留作隐式的依赖，就是被你丢掉的影响
分析。

## 经验法则

写下最少数量的子句，让系统的*承重*承诺变得可判定。
让高风险路径把粒度拉向自身，让其余部分保持为说明性文字，
并把每个子句绑定到你会信任陌生人写出的最强 oracle。拿不准时，
就问[采用与
边界](05-adoption-and-limits.md)里的证伪问题：写这个 oracle 是否比评审
它守护的代码更便宜？如果不是，它可能根本不值得成为子句。
