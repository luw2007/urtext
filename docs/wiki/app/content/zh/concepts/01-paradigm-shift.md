# 范式转变

软件开发正在经历一段迁移序列，这在如今每一个
AI 辅助代码库中都已经清晰可见：

```text
hand-coding → conversational AI → vibe coding → humans stop reviewing code line by line
```

每一步都在移动同一条边界：**人类究竟对什么
负责。** 在手写代码时，你拥有每一行。使用对话式 AI 时，你拥有
提示词并粗略检查输出。在 vibe coding 中，你拥有结果，并完全不再阅读
diff。瓶颈跟着这条边界移动——而它已经移动了。
困难之处不再是*编写*代码。困难之处是*评审*代码。

## 工作对象上移

Urtext 认真对待这场迁移，并为新的工作对象命名：**人类
的工作对象不再是代码——而是对系统的认知。**

这种认知不是单一产物。Urtext 的愿景（VISION §1，原则 P7）
是它横跨四种同等重要的一等载体，它们都属于同一个认知
单元，并彼此交叉引用：

- **规范** 描述行为——系统必须做什么。
- **设计** 描述外观——它必须看起来像什么。
- **交互演示** 描述感受——它必须如何响应。
- **清单** 描述验收——你如何知道它已经完成。

在已交付的 v0 中，第一项和最后一项是真实存在的——Urtext 会发现
Markdown 子句(clause)
文件及其同级的 `tasks.md`，并且 `refs` 将子句链接到子句。设计和
交互载体（以及会检查它们的视觉/交互 oracle(判据)）
是 P7 愿景，目标是 v1，并非今天已经存在（见[这条
边界](../guides/05-adoption-and-limits.md)）。范式主张说的是工作正在去向何处；
当前工具链承载的是行为与验收这两半。

人类维护这些。AI 维护**投影**：代码。代码
成为事实，就像汇编是事实一样——真实、可执行，但不再是
一个人用来理解系统意图的东西。

## 为什么“更好的提示词”不是答案

面对这场迁移，一个诱人的回应是写出更精确的指令——
更丰富的提示词、伪代码、精巧模板。Urtext 直截了当地拒绝这一点。

伪代码和提示词与代码属于同一类别：它们描述机器应该**如何**
行为。增加更多这类东西只会增加歧义，并且会
丢掉代码唯一的优势——它确实会运行。真正的范式
转变并不是提升描述的*精度*。它改变的是
描述的*对象*：从“机器做什么”转向“系统
必须满足什么”。

同样的迁移已经在狭窄领域中赢过三次——数据领域的 SQL、
基础设施领域的 Terraform、正确性领域的类型系统。在每一种情况下，人类
停止编写机制，开始声明约束，而一个
确定性引擎在底层生成并检查机制。

## 成功是什么样子

当工作对象已经完全上移后，人类只评审四件事：

1. **意图变更** —— 规范 diff。
2. **影响** —— 一次改动会波及什么。
3. **验收证据(evidence)** —— 检查是否通过。
4. **争议** —— 模型或证据在哪里出现分歧。

其他一切——满足一个已经决定的意图的常规代码——都
不再逐行阅读。不是因为盲目信任它，而是因为
现在有别的东西来判定它是否正确。这个“别的东西”正是
[下一个概念](02-assembly-to-c.md)的主题。
