# DWARF 映射

这个名字是有意借用的。DWARF 是一种调试信息格式，它
把编译后的机器码缝合回产生它的源码——这样调试器
就能向你显示源码行，而不是原始地址。Urtext 的 DWARF 层旨在
为 AI 时代的抽象实现同一目标：缝合 **子句(clause) ↔ 代码**，让人可以
问“哪条子句约束了这一行？”并得到答案。

这是[事实源
翻转](../concepts/05-source-of-truth-flip.md)的操作核心。

> **v0 范围。** 本页描述今天已经交付的内容：子句→代码范围存储、
> 经 diff 验证的溯源、未映射改动检测，以及*手动* `blame`
> 查询。自动的失败到子句归因——oracle(判据) 或 CI 失败
> 在没有人运行 `blame` 的情况下解析为“违反了 C001”——是设计
> 目标（VISION §2，条件 4），**尚未**接入 verifier。verifier
> 会命名它正在迭代的子句；它不会查询代码映射。

## 映射表

`urtext map` 会把一条子句→代码映射记录到 `clause_code_map(kind, spec_path,
clause_id, file_path, line_start, line_end, commit_sha, note)`。两种 kind 存在于
同一张表中：

- `kind=clause` — 这段代码范围受这条子句约束。
- `kind=ack` — 这个范围是一个*有意未映射*的改动，并已显式
  附带理由确认。

注意，这张表只存储子句和代码。没有证据(evidence)外键——
“子句 ↔ 代码 ↔ 证据”三元组是这一层的抱负，但 v0 持久化的是
子句↔代码这一半。

## 溯源信任 diff，绝不信任自我报告

下面这个机制让翻转变得可强制执行，而不只是寄希望于自觉。当
`map` 或 `ack` 记录一个范围时，该范围必须**与当时实际
`git diff --unified=0 HEAD` 中同一文件的某个 hunk 重叠**。
如果声称映射的行没有发生改动，就会被拒绝。

要精确区分它证明了什么、没有证明什么：

- **它证明**声称范围内*至少有一行*与该文件当前的
  hunk 重叠——你不能针对一个没人碰过的文件记录映射。
  它**不会**验证整个声称范围都发生了改动：声称第 1–100 行
  的范围，只要第 50 行在一个 hunk 中就会通过。
- **它不证明**改动*满足*该子句，甚至不证明这条
  特定子句与该改动相关——一条无关但仍然存在的子句，只要有
  重叠范围，也会被接受。而且下游没有任何东西重新检查
  子句→代码映射的正确性：[元审计(meta-audit)](06-meta-audit-gate.md)会读取
  子句文本、oracle 和证据输出，但从不读取代码映射或 diff，因此
  映射的语义正确性在 v0 中就是**未检查的**。

因此，溯源击败的是*伪造的*映射（针对一个没人
碰过的文件提出的声明），这是对事实源
翻转最重要的失效模式。它是弱重叠的溯源检查，而不是正确性检查。

## 未映射改动强制执行

`urtext check --diff` 会把每个**已跟踪**的工作树 hunk 归因到以下
三者之一：

1. 命中**当前** HEAD 的子句映射，或
2. 显式的 `ack`，或
3. 位于 `specs/<feature>/*.md` 内的位置（规范写回会自我归因）。

其他任何情况都是 `unmapped`，命令会以非零退出：

```text
  ⚠ unmapped src/foo.ts:12-18 — map to a clause, ack, or write back to spec
```

手工编辑并未被禁止。它们会被**显现出来**，并且**必须
回流**：代码 hunk 必须由 `map` 或 `ack` 归因，而一个 hunk 如果
自身路径是 `specs/<feature>/` 下的规范 Markdown 文件（包括 prose 和
`tasks.md`），就会自动自我归因——把意图写回规范本身就是
它自己的回流。这就是事实源翻转的强制执行点。一个
诚实的限制是：它作用于 `git diff HEAD`，而后者只看见**已跟踪**改动。一个
全新的*未跟踪*文件对 `check --diff`（以及 `gate --diff`）都是
不可见的，直到它被 `git add`——所以强制执行覆盖的是对已跟踪代码的编辑，而不是
未跟踪文件的出现。

## Blame 反向运行——手动

`urtext blame <file>:<line>` 回答反向问题——哪条子句
约束了这一行代码。它是一个**手动查询**：你运行它，它读取
代码映射，然后报告。未映射的行会诚实说明：

```text
$ urtext blame src/verifier.ts:1
No clause constrains src/verifier.ts:1.
```

当映射存在时，`blame` 会命名子句。两个 v0 限制：它会返回**每一条
已存储映射**，只要其范围与该行重叠——包括针对旧 commit 记录的映射
以及后来已被删除的子句——没有当前 HEAD 或
存活子句过滤，所以要检查报告的 `commit_sha` 来判断某条映射
是否仍然适用。而且归因是由人运行 `blame` 完成的，不是 verifier
自动完成的。

## v0 边界

范围锚定到 `(file, lines, commit_sha)`，其中 `commit_sha` 是
**diff 基线**——工作树 diff 所基于的变更前 HEAD。
被映射的行是该 diff 的*新*侧，在那个
commit 中尚不存在；SHA 记录的是它们从什么内容 diff 而来，而不是包含
它们的 commit。如果后续编辑移动了已经映射的行，v0 **不会**自动重新锚定
漂移后的范围——这会在后续版本中实现。这是一个
有意标记的简化，不是疏忽。

当意图和代码被缝合在一起后，最后的问题是*由谁决定*——以及
机器如何把人的注意力缩小到只剩真正需要它的地方。这就是
[元审计与裁决门](06-meta-audit-gate.md)。
