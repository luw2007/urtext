# Urtext 隐喻

这个名字不是装饰。它是对系统所做之事目前最精确的
可用描述，借自一个数百年前就解决了结构上相同
问题的学科。

## 乐谱的 ur-text

在古典音乐出版中，*Urtext* 版本是对作曲家原始意图的学术性重建
——是一份剥离了数代编辑“改进”的乐谱。
几十年来，编辑会添加原稿中从未存在过的指法、力度
和乐句处理。Urtext 版本是每一次诚实演出都必须回应的
单一权威来源。

它与由 AI agents 构建的软件之间的映射是精确的：

| Urtext 出版实践 | 本系统 |
|---|---|
| ur-text 是唯一权威；演出是一种诠释 | 规范是真相源；代码是投影 |
| 每次演出都可能不同 | LLM 随机性——同一规范每次运行都会生成不同代码 |
| 是否跑调是可判定的 | oracle(判据) |
| 批判性校勘装置引用每一种读法的依据 | 证据(evidence)加溯源 |
| 编辑性增补必须标记为不同于手稿 | 未映射改动强制执行 |
| 校勘多份手稿以确立文本 | 跨模型元验证 |
| 被数代编辑损坏的乐谱 | 规范腐化 |

## 角色遵循这个隐喻

一旦产物是一份乐谱，演员阵容就很明显：

- **指挥是人类。** 他们阅读完整乐谱，并裁定
  演出是否忠实——他们不演奏乐器。
- **演奏者是 AI agents。** 他们把乐谱实现为声音。
- **演出是代码。** 真实、具体、每次都不同。
- **音叉是 oracle。** 它是不经争论就能决定
  一个音是否准的外部参照。

注意指挥*不*做什么：演奏。人类不在回路中
编写代码。人类在回路中，是为了对照权威来源决定
投影是否忠实——并且只在音叉
沉默（`manual` 子句(clause)）、risk(风险级别)很高（`risk:high`）或
演奏者意见不一致（元审计(meta-audit)争议）时被拉入。

## 为什么产品是乐谱，而不是指挥

这个隐喻里藏着一个微妙的命名教训。产品是**放在
指挥谱架上的 ur-text——而不是指挥。** 工具的命名依据是*它是什么*，而不是
*谁在使用它*。

这就是为什么前五轮候选名称都被拒绝：*tenet, oath,
seal, score, conductor,* 以及整个音乐家家族。“Contract / orchestration
/ score”是编程语言作者的第一反应词汇，而且
三代工具已经占据了那个语义场。好名字
活在外一层——在*载体、仪式、凭证*里——而不在
核心概念词本身里。**Urtext** 是凭证：每一种诠释都必须回应的
权威来源。

标语把这一点说透，让这个借来的词自带注释：*你的系统的
ur-text。代码只是一种诠释。*
