# 事实源翻转

当 C 取代汇编时，有一个文化变化比语法本身做了更多
工作：**手动编辑编译器输出变成了禁忌。** 你修改 C
源码并重新编译。直接编辑生成的汇编不再是
捷径——那是一个等待发生的 bug，因为源码不再描述
实际运行的内容。

Urtext 将其称为**事实源翻转**，而它是六个
[assembly-to-C conditions](02-assembly-to-c.md) 中最难在 AI 时代重现的一个——
因为没有任何东西阻止开发者直接编辑代码，并且永远
不告诉规范。

## Spec Kit 留下的开放问题

Spec Kit 在这一点上非常诚实。它的规范持久化模型为
需求变化后 `spec.md`、`plan.md` 和 `tasks.md` 会发生什么命名了
三种策略：

- **Flow-back**——编辑任意产物，然后手动调和整组产物。
- **Flow-forward**——冻结已完成的产物；创建新的功能目录
  来承载新需求。
- **Living spec**——先编辑 `spec.md`；重新生成派生产物。

然后它把决定性的话明说出来：

> “该模型是一种团队约定，而不是 CLI 设置。”

这是清醒的承认，不是缺陷。但这意味着防御规范
腐化靠的是*纪律*——而这三种模型中的每一种都把自己的失败
模式列为“沉默漂移”、“重复上下文”或“丢失理由”。当团队
忙起来时，约定就是最先被放弃的东西。

## Urtext 的回答：强制执行，而非约定

Urtext 采用 Spec Kit 描述的同样三种时间姿态——它把
已完成修订保留为不可变历史（类似 flow-forward），并通过
引用图传播变更（类似 living spec）——但它拒绝把
翻转留给约定。三个机制强制执行它：

1. **未映射改动检测。** `urtext check --diff` 扫描每个*已跟踪的*
   工作树 hunk。无法归因到子句(clause)映射、显式确认或
   规范写回的改动会被标记为 `unmapped`，并且
   命令以非零退出。（它读取 `git diff HEAD`，所以一个全新的*未跟踪*
   文件会在被 `git add` 之前漏过去——这是 v0 的盲点。）手动编辑
   并未被禁止——它们会被变得*可见*，并且*必须回流*。
2. **溯源相信 diff，而不是 LLM。** 当 agent 声称“这段代码
   满足子句 C001”时，Urtext 不会相信它的一面之词。声称的 clause→code
   范围必须与当时*实际* `git diff` 中的真实 hunk 重叠，否则
   会被拒绝。这至少证明了一条声称的行确实发生了变化——它
   击败了针对未触碰文件的*伪造*映射。它并不证明
   整个范围都变了，也不证明该改动满足子句，甚至不证明这个
   子句与之相关；映射正确性在 v0 中未被检查（
   [元审计(meta-audit)](../mechanisms/06-meta-audit-gate.md) 读取证据，而不是代码
   映射）。溯源奠定的是*文件中有某些东西发生了变化*，而不是*它
   是否正确*。

3. **归因向后运行——v0 中由人工完成。** `urtext blame <file>:<line>`
   将代码行映射回约束它的子句，因此失败可以被
   追溯到意图，而不是堆栈帧。在 v0 中，这是由人运行
   `blame`；把验证器(verifier)接线成自动归因失败是设计
   目标（[DWARF](../mechanisms/05-dwarf-mapping.md)），尚未发布。没有这个
   反向链接，人类就会被迫回到代码层。

## 为什么约定从来不够

原因不是团队没有纪律。它是结构性的：约定没有
失败信号。当有人编辑代码却忘了规范时，不会有任何东西
变红。漂移在构造上就是沉默的，而沉默漂移会不断复合，直到
没有人知道该相信哪个产物。

强制执行提供了缺失的信号。未映射 hunk 是非零退出码
——这是一种强硬、无法忽视的失败，与“一旦编译器可信，
就不要手动编辑汇编”所变成的失败同类。这就是全部诀窍：**让你想
阻止的事情产生失败，你就不再需要每个人都
记得不要去做。**

这三个机制的具体实现——`clause_code_map` 表、
diff 交叉验证和 `urtext blame`——记录在 [DWARF
Mapping](../mechanisms/05-dwarf-mapping.md) 中。
